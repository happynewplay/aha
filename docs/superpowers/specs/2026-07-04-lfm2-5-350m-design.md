# LFM2.5-350M 设计说明

日期：2026-07-04
主题：为 `aha` 添加 `LFM2.5-350M` 模型推理支持

## 1. 背景与目标

当前 `aha` 已支持多种本地模型，并通过统一的 `WhichModel`、`LoadSpec`、CLI `run/serv`、以及 OpenAI 风格 chat completion 接口提供推理能力。现需新增本地模型：

- `D:\model_download\LFM2.5-350M`

本次目标：

1. 在 `aha` 中新增 `lfm2.5-350m` 模型标识。
2. 将模型接入 CLI、registry、服务端与 OpenAI 风格 chat completion 接口。
3. 本次只实现 `safetensors` 运行时。
4. 在 artifact 设计上预留 `gguf` 与 `onnx`，但不在本次实现推理。
5. 支持普通文本 completion 与 stream completion。
6. 对外支持标准 OpenAI 工具调用语义，内部兼容 `LFM2.5` 的原生 tool-call 标签格式。

## 2. 已确认范围

### 用户确认

- 本次范围选择：`C`
  - 正式进入 `WhichModel` 与 CLI/API
  - 暂不接入下载管理
- artifact 选择：`B`
  - 本次运行时只实现 `safetensors`
  - `gguf` 与 `onnx` 只预留框架边界
- tool calling：改为支持标准 OpenAI 工具调用
  - 对外返回标准 `tool_calls`
  - 内部解析 `LFM2.5` 原生标签

### 明确非目标

本次不做：

1. `gguf` 推理运行时
2. `onnx` 推理运行时
3. 图像、视频、音频输入
4. 模型下载管理接入
5. 面向所有 Liquid 模型的通用抽象

## 3. 模型事实与架构判断

根据本地目录内容：

- 存在 `config.json`
- 存在 `model.safetensors`
- 存在 `chat_template.jinja`
- 存在 `tokenizer.json`
- 存在 `tokenizer_config.json`

`config.json` 关键字段：

- `architectures = ["Lfm2ForCausalLM"]`
- `model_type = "lfm2"`
- `hidden_size = 1024`
- `intermediate_size = 6656`
- `num_hidden_layers = 16`
- `num_attention_heads = 16`
- `num_key_value_heads = 8`
- `layer_types = ["conv", "conv", "full_attention", ...]`
- `conv_L_cache = 3`
- `rope_parameters.rope_theta = 1000000.0`
- `tie_embedding = true`
- `eos_token_id = 7`
- `dtype = "bfloat16"`
- `vocab_size = 65536`

由此可知：

1. 该模型不是现有 `MiniCPM5` 所使用的 Llama 变体。
2. 该模型也不应强行并入 `Qwen3` 或 `Qwen3.5` family。
3. 该模型属于独立的 `lfm2` 混合架构，层类型中同时存在 `conv` 和 `full_attention`。
4. 本次应新增独立 family，而不是重命名或复用现有 family。

## 4. 方案比较

### 方案 A：并入现有 `MiniCPM5` 或 `Qwen3` family

优点：

- 改动面小

缺点：

- 模型架构不匹配
- 会让运行时语义混乱
- 后续维护成本高

结论：不采用。

### 方案 B：新增独立 `lfm2_5` family，仅实现 `safetensors` 运行时，并为 `gguf` / `onnx` 预留 artifact 边界

优点：

- 与模型真实架构一致
- 与用户确认范围一致
- 模块边界清晰
- 后续扩展 `gguf` / `onnx` 时不需要推翻本次设计

缺点：

- 需要新建一套 family

结论：采用该方案。

### 方案 C：先抽“混合层文本模型通用框架”，让 `LFM2.5` 与 `Qwen3.5` 共用

优点：

- 长期看更统一

缺点：

- 当前证据不足，容易过早抽象
- `conv`、attention、缓存和权重命名未必可共用
- 会扩大本次实现范围

结论：本次不采用。

## 5. 模块设计

新增目录：

- `src/models/lfm2_5/mod.rs`
- `src/models/lfm2_5/config.rs`
- `src/models/lfm2_5/model.rs`
- `src/models/lfm2_5/generate.rs`
- `src/exec/lfm2_5.rs`

职责划分：

### `config.rs`

负责反序列化 `config.json`，至少包含：

- `bos_token_id`
- `eos_token_id`
- `pad_token_id`
- `dtype`
- `hidden_size`
- `intermediate_size`
- `max_position_embeddings`
- `num_attention_heads`
- `num_hidden_layers`
- `num_key_value_heads`
- `layer_types`
- `conv_L_cache`
- `rope_parameters`
- `tie_embedding`
- `use_cache`
- `vocab_size`

### `model.rs`

负责 `LFM2.5-350M` 模型本体：

- `Lfm2_5Model::new_from_vb(...)`
- `forward(...)`
- `clear_cache()`

内部按 `layer_types` 构建两类层：

- `Conv`
- `FullAttention`

### `generate.rs`

负责统一推理入口：

- `init_from_spec(...)`
- `init(...)`
- `generate(...)`
- `generate_stream(...)`

同时负责：

- chat template 加载
- tokenizer 加载
- eos 停止判断
- 标准 OpenAI tool-call 解析与组装

### `exec/lfm2_5.rs`

为 CLI `run` 提供直接运行封装，行为与现有文本模型保持一致。

## 7. Artifact 设计

### 声明层

在 `supported_artifacts(WhichModel::LFM2_5_350M)` 中声明：

- `Safetensors`
- `Gguf`
- `Onnx`

默认 artifact：

- `default_artifact(WhichModel::LFM2_5_350M) -> Safetensors`

### 运行时层

在 `registry::load_model_from_spec(...)` 进入 `LFM2_5` family 后：

- `Safetensors`：正常初始化
- `Gguf`：返回明确错误
- `Onnx`：返回明确错误

建议错误信息：

- `lfm2.5-350m gguf runtime is not implemented yet`
- `lfm2.5-350m onnx runtime is not implemented yet`

这样可以保证：

1. CLI 参数和 artifact 框架稳定
2. 本次未实现的格式不会静默失败
3. 后续扩展时无需变更外部接口

## 8. Safetensors 路径设计

本次仅支持本地 `safetensors` 目录加载，目录依赖：

- `config.json`
- `model.safetensors` 或目录内其他 `*.safetensors`
- `chat_template.jinja`
- `tokenizer.json`
- `tokenizer_config.json`

初始化流程：

1. 使用 `ChatTemplate::init(path)` 加载模板
2. 使用 `TokenizerModel::init(path)` 加载 tokenizer
3. 读取 `config.json`
4. 使用 `find_type_files(path, "safetensors")` 查找权重
5. 使用 `VarBuilder::from_mmaped_safetensors(...)` 构造权重访问器
6. 调用 `Lfm2_5Model::new_from_vb(...)` 初始化模型

## 9. 推理行为设计

### 输入与输出

本次仅支持：

- 文本 chat completion
- 文本 stream completion

不支持：

- 图像输入
- 视频输入
- 音频输入

### 终止条件

终止 token 使用配置中的：

- `eos_token_id = 7`

### 采样策略

与现有文本模型保持一致：

- 复用 `get_logit_processor(...)`
- 支持 `temperature`
- 支持 `top_p`
- 支持 `seed`

### Chat Template

使用模型目录中的 `chat_template.jinja`，不为 `LFM2.5` 单独手写模板拼接逻辑。

理由：

1. 本地模型目录已提供模板
2. 模板与模型特殊 token 对齐
3. 可复用现有 `ChatTemplate` 机制

### Stream 行为

stream 路径：

- 逐 token 生成
- 复用现有 UTF-8 半截字符缓冲策略
- 输出 `build_completion_chunk_response(...)`

### Tool Calling

本次对外支持标准 OpenAI 工具调用语义。

内部模型原生边界仍按 `LFM2.5` 的特殊 token 识别：

- `<|tool_call_start|>`
- `<|tool_call_end|>`

解析与输出策略：

1. 当请求携带 tools 时，继续通过现有 chat template 机制把工具定义注入 prompt。
2. 模型若输出 `<|tool_call_start|>...<|tool_call_end|>` 包裹的函数调用内容，运行时负责将其解析为标准 OpenAI `tool_calls` 结构。
3. 非 stream 路径返回的 `ChatCompletionResponse` 中，assistant message 使用标准 `tool_calls` 字段承载函数调用，而不是把原始标签文本直接暴露给调用方。
4. stream 路径输出的 chunk 同样使用标准 OpenAI `tool_calls` 增量字段。
5. 标签外的普通文本继续作为 assistant 文本内容输出。

兼容边界：

- 本次只保证从 `LFM2.5` 原生标签格式到标准 OpenAI `tool_calls` 的单向适配。
- 不要求模型原生直接产出 JSON 版 OpenAI response。
- 若 tool-call 内容不完整或解析失败，优先降级为普通文本输出，并避免返回畸形 `tool_calls`。

## 10. 底层架构设计

本次按“独立文本模型”实现，不复用 `MiniCPM5` 的 Llama 主体，也不将内部结构抽象到全局通用层。

### 结构划分

建议核心结构：

- `Lfm2_5Config`
- `Lfm2_5Layer`
- `Lfm2_5TextModel`
- `Lfm2_5Model`

#### `Lfm2_5Layer`

层类型为枚举：

- `Conv`
- `FullAttention`

按 `config.layer_types` 顺序构建全部 16 层。

#### `Lfm2_5TextModel`

包含：

- `embed_tokens`
- `layers`
- `final_norm`
- `lm_head`

若 `tie_embedding = true`，则 `lm_head` 复用 embedding 权重。

### 缓存策略

- `FullAttention` 层维护 KV cache
- `Conv` 层维护短窗口状态缓存
- `Conv` 层缓存窗口长度来自 `conv_L_cache`
- 每次请求结束后统一 `clear_cache()`

这样可以避免：

- 服务端跨请求串状态
- stream 与非 stream 混用后残留缓存

### 抽象边界

本次不将 `Conv` 组件上提到 `src/models/common`

原因：

1. 当前仅此模型需要该结构
2. 过早抽象风险高
3. 后续若出现第二个同类模型，再决定是否抽公共组件

## 11. 错误处理设计

需要显式处理以下错误：

1. `config.json` 缺失或字段不完整
2. `chat_template.jinja` 缺失
3. tokenizer 文件缺失
4. 目录中找不到任何 `*.safetensors`
5. `layer_types` 中出现未知值
6. 请求以 `gguf` 或 `onnx` 方式加载时，运行时未实现
7. tool-call 标签未闭合或内容无法解析为有效函数调用

处理原则：

- 错误信息要包含缺失文件名、缺失字段名或未实现格式名
- 对未闭合或无法解析的 tool-call，不抛致命错误，优先降级为普通文本
- stream 结束时安全清空 tool-call 缓冲与模型缓存

## 14. 最终设计结论

本次采用：

- 新增独立 `lfm2_5` family
- 正式进入 `WhichModel`、CLI、API、registry
- 本次只实现 `safetensors` 运行时
- `gguf` / `onnx` 只预留 artifact 与错误边界
- 对外支持标准 OpenAI 工具调用
- 内部兼容 `LFM2.5` 风格 tool-call 标签解析
- 不接入下载管理
- 不做无关的全局抽象重构

该设计与当前代码库的模块边界一致，能以可控改动完成 `LFM2.5-350M` 首次接入，并为后续 `gguf` / `onnx` 运行时扩展保留稳定接口。
