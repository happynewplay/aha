# LFM2.5-230M 实现计划

> **面向 AI 代理的工作者：** 在当前工作树中以内联方式执行此计划。使用复选框（`- [ ]`）跟踪步骤；不创建分支、工作树或提交。

**目标：** 将 `lfm2.5-230m` 注册为 LFM2.5 系列模型，并复用既有 Safetensors、GGUF 和 ONNX 加载器。

**架构：** `WhichModel::LFM2_5_230M` 承担命令行/API 模型身份。制品能力、注册器和 CLI 分发把该变体归入现有 LFM2.5 分支，模型结构继续由所选权重目录中的 `config.json` 驱动。

**技术栈：** Rust 2024、Clap `ValueEnum`、Candle、Cargo 测试。

---

## 文件结构

- 修改：`src/models/mod.rs`，声明 LFM2.5-230M 并提供模型元数据。
- 修改：`src/models/core/artifact.rs`，为 230M 声明与 350M 相同的制品能力和默认格式。
- 修改：`src/models/core/registry.rs`，将 230M 路由到既有 `LFM2_5` 加载器。
- 修改：`src/exec/lfm2_5.rs`，由加载规格生成默认请求中的模型名。
- 修改：`src/main.rs`，让 CLI 分发和 `--request-json` 支持整个 LFM2.5 文本系列。
- 修改：`tests/test_load_spec.rs`，覆盖公开模型名解析、格式能力、默认格式和 `LoadSpec` 验证。
- 修改：`modify_test.md`，记录唯一受影响的集成测试与单测试命令。

```rust
#[test]
fn load_spec_accepts_lfm2_5_230m_artifacts() {
    let model = WhichModel::from_str("lfm2.5-230m", true).unwrap();

    assert_eq!(model.openai_model_id(), "lfm2.5-230m");
    assert_eq!(model.owner(), "LiquidAI");
    assert!(!model.is_download_managed());
    assert_eq!(default_artifact(model), ArtifactKind::Safetensors);
    assert_eq!(
        supported_artifacts(model),
        &[ArtifactKind::Safetensors, ArtifactKind::Gguf, ArtifactKind::Onnx]
    );
    assert!(LoadSpec::for_safetensors(model, "D:/model_download/LFM2.5-230M")
        .validate()
        .is_ok());
}
```

### 任务 2：将 230M 纳入既有 LFM2.5 系列

**文件：**
- 修改：`src/models/mod.rs`
- 修改：`src/models/core/artifact.rs`
- 修改：`src/models/core/registry.rs`
- 修改：`src/exec/lfm2_5.rs`
- 修改：`src/main.rs`

- [ ] **步骤 1：添加模型身份和元数据**

在 `src/models/mod.rs` 中紧邻 350M 变体添加：

```rust
#[value(name = "lfm2.5-230m", hide = true)]
LFM2_5_230M,
```

将变体加入 `LISTED_MODELS`，并在既有方法中与 350M 保持同组行为：

```rust
WhichModel::LFM2_5_230M => "lfm2.5-230m",
```

同时将其加入本地管理排除项、`LiquidAI` 所有者映射和 `llm` 模型类型映射。

- [ ] **步骤 2：复用 350M 制品能力**

在 `src/models/core/artifact.rs` 中将 `WhichModel::LFM2_5_230M` 与
`WhichModel::LFM2_5_350M` 归入同一匹配分支，使两者均支持
`Safetensors`、`Gguf`、`Onnx`，默认采用 `Safetensors`。

- [ ] **步骤 3：复用 LFM2.5 注册加载器**

在 `src/models/core/registry.rs` 的 `resolve_model_loader_family` 中，把
`WhichModel::LFM2_5_230M` 与 `WhichModel::LFM2_5_350M` 合并映射到
`ModelLoaderFamily::LFM2_5`。不修改 `Lfm2_5GenerateModel`，因为它已从权重目录的 `config.json` 加载模型尺寸。

- [ ] **步骤 4：同步 CLI LFM2.5 入口**

在 `src/main.rs` 中将 230M 加入 LFM2.5 的 `--request-json` 允许列表、`run_target_model_with_spec` 分发分支和后续已处理模型分支。在 `src/exec/lfm2_5.rs` 中将默认请求的模型名改为 `spec.model.openai_model_id()`，避免 230M 的响应请求仍标为 350M。
