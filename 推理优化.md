在纯 CPU(含核显但不走 GPU 后端)场景下用 Candle 推理,能拿到收益的优化点大致分这几层:

## 1. 编译期优化(收益最大,做起来最简单)

- **开启 MKL / Accelerate**:Candle 的 CPU matmul 默认走自己写的 gemm(基于 `gemm` crate),但如果开 `mkl` feature(x86)或 `accelerate` feature(macOS),会走 Intel MKL / Apple Accelerate,矩阵乘法能快数倍。
```toml
candle-core = { version = "*", features = ["mkl"] }
```
- **RUSTFLAGS 开启目标 CPU 全部指令集**,尤其 AVX2/AVX512/FMA:
```bash
RUSTFLAGS="-C target-cpu=native" cargo build --release
```
这一点经常被忽略,但对 `gemm` crate 走的向量化路径影响巨大,因为默认编译不会用你 CPU 实际支持的所有指令集。

- 确认 release 模式 + `lto = true` + `codegen-units = 1`(Cargo.toml profile.release),减少间接调用开销。

## 2. 量化

- Candle 支持 GGUF/量化权重(Q4_0、Q4K、Q8_0 等),CPU 上量化模型不仅省内存带宽,速度也明显更快,因为纯 CPU 推理很多时候是内存带宽瓶颈而不是算力瓶颈。
- 优先用 Q4_K_M / Q5_K_M 这类分组量化格式,比朴素 Q4_0 精度损失小、速度接近。
- 如果你已经在做 Ascend/vLLM 上的 w8a8 量化对比,CPU 上思路类似:量化本质是减少内存搬运。

## 3. 线程与并行策略

- Candle 底层 gemm 会用 `rayon` 并行,线程数默认等于逻辑核数,但**超线程开太多反而会因为 cache 争用变慢**,建议显式设置为物理核数:
```rust
std::env::set_var("RAYON_NUM_THREADS", "8"); // 按物理核数调
```
- 如果你的 CPU 是异构架构(P-core/E-core,比如新版 Intel),把线程绑定在 P-core 上(用 `core_affinity` crate)通常比让调度器乱跳收益更大。
- NUMA 多路服务器要注意:跨 NUMA 节点访问内存代价很高,单进程推理尽量绑定在一个 NUMA node 内(`numactl --cpunodebind=0 --membind=0`)。

## 4. 内存带宽相关

- CPU 推理大模型时瓶颈往往是内存带宽而非算力,所以:
  - 尽量用量化权重减小体积;
  - 确认内存是双通道/四通道全部插满,单通道内存会直接腰斩吞吐;
  - `mmap` 加载权重(Candle 的 safetensors 加载默认支持 mmap),避免一次性拷贝进内存造成额外开销和冷启动慢。

## 5. KV Cache 与 Attention 实现

- 确保用的是 Candle 里针对 CPU 优化过的 attention 实现(不同模型仓库实现质量差异很大,有些第三方 Candle 模型移植代码没有做 KV cache 复用,每步都重算全部 attention)。
- 如果是自己写推理循环,检查 KV cache 是否真正做了增量更新,而不是每个 token 重新 forward 整个序列。

## 6. Batch / Prefill 优化

- Prefill 阶段(prompt 处理)是纯矩阵乘法密集型,MKL/AVX512 收益最明显;
- Decode 阶段(逐 token 生成)是内存带宽密集型、matmul 很小,这时候线程数不是越多越好,反而线程调度开销可能超过计算本身——可以对 decode 阶段单独调低并行度。

## 7. 核显(iGPU)方面

Candle 目前 CPU 后端不会自动利用核显,除非你显式用它的 Metal(Apple 核显)或未来的 Vulkan/OpenCL 后端。如果你想真正利用核显算力,现状是:
- Apple 芯片:用 `metal` feature,收益最大;
- Intel/AMD 核显:Candle 原生支持有限,更现实的路径是用 llama.cpp 的 SYCL/Vulkan 后端做核显推理,Candle 目前这块还不成熟。

13400 是 10 核(6P+4E)16 线程,注意几个关键点:

## 1. AVX-512 不可用
Raptor Lake 因为有 E-core,Intel 把 AVX-512 在 BIOS/微码层面关掉了(即使 P-core 硬件支持)。所以:
- `target-cpu=native` 编译时不会启用 AVX-512 路径,实际会退到 AVX2,这是预期行为,不用纠结怎么强开(强开也用不了,E-core 完全不支持)。
- 走 AVX2 + FMA 已经是这颗 CPU 能拿到的上限,MKL 在 AVX2 上依然比默认 gemm 快,该开还是要开。

## 2. 线程数——这是 13400 上最容易踩坑的地方
- **不要设成 16**(逻辑线程数)。E-core 没有超线程,只有 4 个 P-core 有超线程,混着用调度会很乱。
- 推荐先只用 **P-core 的 6 个物理核**(不算超线程):
```bash
RAYON_NUM_THREADS=6
```
- 更进一步,用 `core_affinity` 或者 `taskset` 把进程绑定在 P-core 上,避免 Windows/Linux 调度器把线程甩到 E-core 上拖慢矩阵乘法(E-core 单核吞吐明显低于 P-core,混跑会导致最慢的线程拖累整体同步点):
```bash
# Linux 下先查 P-core 是哪几个逻辑核(通常是 0-11 里的偶数或前几个,视BIOS而定)
taskset -c 0,2,4,6,8,10 ./your_binary
```
具体哪几个编号是 P-core,可以用 `lscpu -e` 看 CPU 的 `MAXMHZ` 区分(P-core 主频标注更高)。

- 如果想把 E-core 也用上(4 个),可以试 `RAYON_NUM_THREADS=10`,但很多时候因为负载不均衡(P-core 快 E-core 慢,gemm 分块是均分任务量),实测反而比只用 6 P-core 慢或持平,建议自己 benchmark 对比一下,别直接信理论峰值。

## 3. 内存
13400 官方支持 DDR4-3200 或 DDR5-4800(看你板子),**务必确认双通道插满**,单通道会直接腰斩带宽,对 CPU 推理这种带宽敏感场景影响比换 CPU 还大。

## 5. 模型规模上的现实预期
13400 这个级别,纯 CPU 跑:
- 1-3B 量化模型(Q4_K_M):体验尚可,能到几 token/s 到十几 token/s
- 7-9B 量化模型:能跑但偏慢,decode 阶段大概几 token/s
- 更大的模型(你提到过在评估的 Qwen3-27B 之类)纯 CPU 上基本不现实,那个还是得放到 Ascend/4090 那边跑

你这台 13400 具体是想跑什么规模的模型做什么用途(比如给 Hive 里做本地小模型 tool-calling,还是别的)?如果是 tool-calling 场景,量化后的 1B 级别模型在 13400 上 CPU 推理延迟应该是完全够用的。


0.3B Q8 只有几 token/s 明显不正常,这个规模在 13400 上应该轻松跑到 50-100+ token/s。说明没吃满硬件,大概率是以下几个问题之一,按可能性排查:

## 1. 编译时是不是 debug 模式(最常见)
先确认:
```bash
cargo build --release
```
如果之前跑的是 `cargo run` 不带 `--release`,debug 模式下 candle CPU 推理可能慢 **10-50 倍**,这是最常见的踩坑点,先排除这个。

## 2. 是不是没开 MKL / gemm 没用向量化
检查 Cargo.toml 里 candle-core 有没有开 `mkl` feature,以及编译时有没有 `RUSTFLAGS="-C target-cpu=native"`。如果两者都没开,默认 gemm 实现在小 batch 场景下效率会很差。

验证方法:跑的时候 `htop` 看 CPU 占用——
- 如果只有 1 个核心接近 100%,其他核心空转 → 说明矩阵乘法根本没并行,rayon 没生效或者线程数被设成 1
- 如果所有核心都在跑但速度还是慢 → 更可能是量化 kernel 效率问题

## 3. Q8 反量化 kernel 有没有针对性优化
Candle 对不同量化格式(Q8_0, Q4_0, Q4K...)的 CPU kernel 优化程度不一样,部分格式在早期版本里是标量实现(没有 SIMD),尤其如果你用的是第三方模型移植代码里手写的量化算子,而不是 candle-core 自带的 `quantized` 模块。

建议确认:
- 是否用的是 `candle-transformers` 里官方维护的模型实现,还是自己/别人写的推理循环
- candle 版本是不是比较旧(早期版本 CPU 量化 kernel 性能差很多,近期版本优化了不少)

## 4. 检查是不是卡在非计算部分
几 token/s 对 0.3B 来说慢到有点像不是纯计算瓶颈,可能是:
- 每个 token 都在做不必要的重复计算(KV cache 没生效,每步重新 forward 整个序列)
- tokenizer 编解码、采样逻辑里有同步 I/O 或者 print/flush 每个 token 都刷屏拖慢
- 用了 `f32` 中间计算精度但没有利用好,或者 attention 实现是 O(n²) 重复重算

## 排查顺序建议

1. 先确认 `--release` + `RAYON_NUM_THREADS=6` + htop 观察核心利用率
2. 用 candle 官方 `candle-examples` 里的量化模型 example 跑一遍同样量化格式的小模型做基准对比,如果官方 example 也慢,说明是环境/编译问题;如果官方 example 快很多,说明是你的推理代码实现问题(大概率是 KV cache 没生效)
